# Mynah Release And Download Specification

## Purpose

This document defines what the website should say about downloading Mynah and how release metadata should be displayed.

The goal is to prevent stale claims like `v0.1.1` appearing without the incrementing release number, or Apple Silicon and universal binary claims appearing at the same time.

---

## Release Identity

Mynah now uses two version concepts:

| Field | Meaning | Example |
| --- | --- | --- |
| Marketing version | Product/version line | `0.1.1` |
| Release number | Incrementing build/release identity from Git history | `Release 103` |
| Bundle version | macOS build identifier, same as release number | `103` |
| Build signature | Full support/debug signature | `0.1.1+r103.ce6fad5f7bcb` |

The website should never show only `v0.1.1` as the visible release identity.

Recommended display:

`Version 0.1.1 · Release 103`

For support or advanced metadata:

`0.1.1+r103.ce6fad5f7bcb`

---

## Download Button Requirements

### Primary button label

Download for macOS

### Button metadata

Use this format:

`Version {marketing_version} · Release {release_number} · {architecture_label} · macOS {minimum_macos}+`

Example:

`Version 0.1.1 · Release 103 · Apple Silicon · macOS 10.15+`

### Do not say

- `Free forever` unless pricing is permanently committed
- `Universal binary` unless the downloadable artifact is truly universal
- `Apple Silicon` if the artifact also contains Intel
- `Intel supported` unless a tested Intel build is actually attached

---

## Architecture Labels

Use one of these exact labels:

| Artifact type | Website label |
| --- | --- |
| Apple Silicon only | `Apple Silicon` |
| Intel only | `Intel Mac` |
| Universal app | `Apple Silicon + Intel` |
| Unknown or not verified | `macOS build` |

The website must not mix labels for the same download.

## Download Section Copy

### Recommended section title

Download Mynah for macOS

### Recommended section copy

Install the latest release, move it to Applications, approve macOS permissions, and start dictating with Fn.

### Install checklist

1. Download the DMG.
2. Open it and move Mynah to Applications.
3. Launch Mynah from Applications.
4. Allow Microphone access.
5. Allow Accessibility access when macOS asks.
6. Hold Fn, speak, and release to paste.

---

## Permission Note For Download Area

Recommended copy:

Mynah needs Microphone permission to capture your voice and Accessibility permission to paste into the active app. If macOS keeps an old Accessibility entry after reinstalling, Mynah attempts to refresh it and guide approval again.

## Release Metadata Source

Current build metadata is generated by:

- `apps/mynah/scripts/prepare-build-meta.mjs`

Generated app metadata includes:

- `marketingVersion`
- `bundleVersion`
- `gitCommitCount`
- `builtAtIso`
- `gitCommit`
- `gitDirty`
- `buildSignature`

The website should ideally read release metadata from a small manifest beside the download artifact instead of hardcoding it into page copy.

Suggested future manifest:

```json
{
  "product": "Mynah",
  "marketingVersion": "0.1.1",
  "releaseNumber": 103,
  "bundleVersion": "103",
  "buildSignature": "0.1.1+r103.ce6fad5f7bcb",
  "architectureLabel": "Apple Silicon",
  "minimumMacOS": "10.15",
  "downloadHref": "public/Mynah_0.1.1_aarch64.dmg",
  "fileSizeLabel": "15 MB"
}
```

## Website Acceptance Checks

- The hero download button and download section show the same artifact and architecture.
- The visible release line includes `Release N`.
- The old standalone `v0.1.1` badge is removed or rewritten.
- The DMG filename matches the actual uploaded artifact.
- Minimum macOS version matches `tauri.conf.json`.
- Architecture label matches the built DMG.
- Support text mentions Microphone and Accessibility permissions.
